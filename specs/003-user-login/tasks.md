- [x] T009 [US1] 在 src/js/app.js 新增工作階段管理模組：checkSession()（檢查 sessionStorage `todo-webapp-session`）、setSession()（設定 `"authenticated"`）、clearSession()（移除鍵值）

## 相依性與執行順序

### 階段相依性

- **設置（階段 1）**：無相依性——可立即開始
- **基礎建設（階段 2）**：相依於設置完成（HTML 結構必須存在才能撰寫 CSS 規則）
- **使用者故事 1（階段 3）**：相依於基礎建設完成（CSS 切換規則必須就緒）
- **使用者故事 2（階段 4）**：相依於使用者故事 1 完成（需要登入表單提交處理已存在）
- **使用者故事 3（階段 5）**：相依於使用者故事 1 完成（需要 setSession/clearSession 函式）
- **收尾（階段 6）**：相依於所有使用者故事完成

### 使用者故事相依性

- **使用者故事 1（P1）**：工作階段管理和憑證驗證為核心——所有後續故事依賴此階段
- **使用者故事 2（P1）**：擴展 US1 的登入處理函式，新增失敗分支——依賴 US1
- **使用者故事 3（P2）**：使用 US1 建立的 clearSession()——依賴 US1，與 US2 無依賴可平行

### 每個使用者故事內部

- 工作階段管理函式先於登入處理
- 憑證驗證函式先於表單事件綁定
- 核心邏輯先於 UI 互動回饋

### 平行執行機會

- 階段 1 的 T001、T002、T003 皆修改同一檔案（index.html），建議循序執行
- 階段 2 的 T004～T008 標記 [P]，皆為 CSS 新增且互不衝突，可平行執行
- 階段 4（US2）和階段 5（US3）皆依賴 US1 完成後可平行開始（US2 因修改同一函式建議在 US3 前執行）

## 實作策略

### 最小可行產品 (MVP) 優先（使用者故事 1）

1. 完成階段 1：在 index.html 新增 HTML 結構
2. 完成階段 2：在 styles.css 新增所有 CSS 樣式
3. 完成階段 3：在 app.js 實作核心登入流程
4. **停止並驗證**：可用 admin/admin 登入、看到歡迎訊息、進入主畫面、重新整理後保持登入

### 漸進式交付

1. 設置 + 基礎建設 → 視覺結構就緒
2. 使用者故事 1 → 核心登入可運作（MVP！）
3. 使用者故事 2 → 錯誤提示完善
4. 使用者故事 3 → 登出功能可運作
5. 收尾 → 防呆、存取保護驗證、完成

每個故事增加價值而不破壞先前的故事。
